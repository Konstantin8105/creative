Техническое задание на разработку препроцессора для редактирования 3D моделей конечных элементов

1. Назначение и цели разработки
1.1. Разработать препроцессор для создания и редактирования 3D моделей конечных элементов, предназначенный для инженерных расчетов.
1.2. Целью разработки является создание программного продукта, обеспечивающего интуитивно понятный ввод и редактирование геометрии, нагрузок и граничных условий для последующего расчета методом конечных элементов.

2. Требования к программному обеспечению
2.1. Требования к аппаратному и системному обеспечению:
   - Операционная система: Windows 10 (полностью нативное поведение).
   - Графическая подсистема: поддержка OpenGL версии 2.1 или 3.0 для рендеринга 3D сцены.
   - Язык реализации: Go (Golang).
2.2. Требования к поддерживаемым типам элементов:
   - Узел (node) - точечный элемент с координатами X, Y, Z.
   - Линия (line) - двухузловой линейный элемент.
   - Треугольник (triangle) - трехузловой поверхностный элемент.
   - 4-узельная пластина (4-node plate) - четырехузловой поверхностный элемент.
2.3. Размерность пространства: трехмерное декартово пространство (координаты X, Y, Z).

3. Функциональные требования

3.1. Архитектура приложения
3.1.1. Реализовать интегрированную среду, позволяющую открывать несколько моделей одновременно в отдельных окнах.
3.1.2. Каждое окно модели должно содержать:
   - Строку заголовка с именем файла текущей модели.
   - Панель быстрого доступа для основных операций (новый файл, открыть, сохранить, настройки приложения).
   - Систему вкладок для переключения между режимами работы.
3.1.3. Строка состояния должна отображать:
   - Общее количество узлов и элементов в модели.
   - Количество выбранных объектов (с выделением цветом).
   - Текущие единицы измерения с возможностью их быстрой смены по клику мыши.
   - Текущие углы обзора камеры с возможностью точной установки.
   - Индикатор прогресса выполнения длительных операций.

3.2. Режимы работы приложения
3.2.1. Предусмотреть следующие основные режимы работы, реализованные как вкладки:
   - Графическое моделирование (основное 3D пространство для работы с моделью).
   - Табличное редактирование (табличный редактор для прямого ввода и редактирования данных).
   - Управление расчетными случаями и комбинациями нагрузок.
   - Работа с материалами и сечениями.
   - Заметки (текстовое поле для хранения комментариев к модели).
   - Сводка по модели (информация для проверки и формирования отчетов).
   - Настройки решателя (подготовка данных для расчета).
   - Просмотр результатов (отображение численных результатов расчета).

3.3. Режим графического моделирования
3.3.1. Управление видом должно обеспечивать:
   - Вращение модели при зажатой левой кнопке мыши и перемещении курсора.
   - Установку новой точки вращения двойным кликом левой кнопки мыши по произвольному месту модели.
   - Масштабирование модели при зажатой правой кнопке мыши и вертикальном перемещении курсора, а также при помощи колеса мыши.
   - Панорамирование модели при зажатых обеих кнопках мыши (левой и правой) и перемещении курсора.
   - Поддержку горячих клавиш для основных команд управления видом.
3.3.2. Режимы отображения при вращении модели должны включать:
   - Каркасный режим (wireframe).
   - Режим ограничивающих параллелепипедов (box).
   - Режим сплошного рендеринга (solid).
3.3.3. Курсор мыши должен динамически изменять свою форму в зависимости от активного инструмента или режима.
3.3.4. Панель инструментов выбора должна обеспечивать:
   - Фильтрацию выбираемых сущностей по типу (узлы, линии, пластины).
   - Выбор прямоугольной областью.
   - Выбор произвольной областью.
   - Выбор всех сущностей.
   - Инверсию текущего выбора.
   - Сохранение и загрузку именованных наборов выбора.
   - Блокировку выбранных или невыбранных сущностей от дальнейших операций.
3.3.5. Панель видимости должна обеспечивать быстрое включение/отключение отображения различных типов элементов и их атрибутов.
3.3.6. Контекстное меню, вызываемое правым кликом в области 3D вида, должно содержать команды создания, редактирования, удаления и просмотра свойств объектов.
3.3.7. Реализовать систему привязок:
   - Универсальные привязки к координатной сетке.
   - Объектные привязки (к узлам, серединам линий, центрам граней и т.д.).
3.3.8. Поддержать различные типы вспомогательных сеток:
   - Ортогональные сетки.
   - Радиальные сетки.
   - Наклонные сетки.
   - Возможность использования DXF подложки в качестве фона.

3.4. Режим табличного редактирования
3.4.1. Данные должны быть организованы по вкладкам для каждого типа сущностей:
   - Узлы (Nodes).
   - Линии (Lines).
   - Треугольники (Triangles).
   - Пластины (Plates).
3.4.2. Внутри каждой вкладки должны быть подвкладки для различных типов данных:
   - Координаты и геометрия.
   - Приложенные нагрузки.
   - Граничные условия (restraints).
   - Связи и жесткости.
3.4.3. Табличное редактирование должно реализовывать функциональность, аналогичную электронным таблицам:
   - Прямое редактирование содержимого ячеек.
   - Копирование и вставка данных через буфер обмена с возможностью обмена с внешними Windows-приложениями (Microsoft Excel, текстовые редакторы).
   - Добавление и удаление строк (сущностей).
   - Сортировка данных по выбранным колонкам.
   - Поиск и фильтрация данных.
   - Выполнение математических операций над диапазонами ячеек.
   - Фильтрация по статусу выбора (отображение только выбранных или только невыбранных объектов).
3.4.4. Обеспечить динамическую синхронизацию между таблицами и графическим представлением:
   - Изменения, внесенные в таблицах, должны мгновенно отображаться в 3D виде.
   - Выбор объектов в 3D виде должен приводить к выделению соответствующих строк в таблицах.
3.4.5. Предусмотреть возможность открытия нескольких таблиц одновременно.
3.4.6. Режимы отображения вкладок:
   - Компактный режим: отображаются только те вкладки и подвкладки, данные в которых реально присутствуют в текущей модели.
   - Расширенный режим: отображаются все возможные вкладки и подвкладки.
3.4.7. Обеспечить возможность выбора системы координат (глобальная или пользовательская) для отображения векторных величин (координаты узлов, векторы нагрузок).
3.4.8. Реализовать древовидную структуру для организации элементов модели в логические группы с возможностью:
   - Создания, переименования и удаления групп.
   - Включения/отключения видимости всех элементов группы.
   - Включения/отключения возможности выбора всех элементов группы.

3.5. Система нагрузок и комбинаций
3.5.1. Реализовать иерархическую древовидную структуру для управления нагрузками, включающую следующие разделы:
   - Первичные загружения (нагрузки).
   - Комбинации загружений.
   - Расчетные ситуации.
3.5.2. Для первичных загружений предусмотреть:
   - Возможность создания загружений с указанием их типа (постоянные, временные длительные, кратковременные, особые, сейсмические).
   - Редактирование, копирование и удаление загружений через контекстное меню.
   - Группировку связанных загружений в папки внутри дерева.
   - Быстрое включение/отключение визуализации нагрузок от конкретного загружения в 3D виде.
   - Отображение сводной информации по нагрузкам и опорным реакциям для каждого загружения.
3.5.3. Для комбинаций загружений реализовать диалоговое окно создания и редактирования со следующими возможностями:
   - Выбор загружений, входящих в комбинацию, с заданием коэффициентов сочетания для каждого загружения.
   - Возможность использования в комбинации не только первичных загружений, но и ранее созданных комбинаций.
   - Выбор типа комбинации в соответствии с различными строительными нормами и правилами (предельные состояния, эксплуатационные, сейсмические).
   - Автоматический подсчет и отображение количества формируемых комбинаций.
   - Опции учета благоприятного или неблагоприятного воздействия постоянных нагрузок.
   - Возможность задания взаимоисключающих загружений (например, ветер по направлению X+ и ветер по направлению X-).
   - Механизм автоматического определения главного переменного загружения для каждой комбинации.
3.5.4. Реализовать генератор комбинаций, обеспечивающий автоматическое создание наборов комбинаций на основе выбранных строительных норм (Еврокоды, нормы ASCE, СНиП, СП). Генератор должен:
   - Учитывать национальные приложения и особенности применения норм.
   - Предоставлять возможность выбора типов формируемых комбинаций.
   - Позволять настраивать параметры генерации.

3.6. Математический аппарат и работа с моделью
3.6.1. Обеспечить поддержку всех систем единиц измерения с возможностью конвертации значений в любой момент времени.
3.6.2. Реализовать неограниченную историю действий (Undo/Redo) для операций, выполняемых как в графическом, так и в табличном режиме.
3.6.3. Предусмотреть механизм автоматического резервного копирования по истечении заданного интервала времени.
3.6.4. Создать инструменты генерации типовых моделей на основе шаблонов:
   - Регулярные сетки и поверхности.
   - Диски, цилиндры, конусы.
   - Арки и ферменные конструкции.
   - Баки и резервуары.
   - Гидростатические нагрузки.
3.6.5. Реализовать набор CAD-подобных инструментов редактирования:
   - Создание, копирование, зеркальное отражение.
   - Перемещение, поворот, масштабирование выделенных объектов.
   - Разделение и слияние элементов.
   - Тримминг (обрезка) линий и поверхностей.
   - Выдавливание (экструзия) профилей.
   - Построение сетки по полигональной области с генерацией качественных четырехугольных элементов.
3.6.6. Поддержать импорт и экспорт распространенных форматов:
   - Импорт геометрии из форматов DXF, IGES, STEP для интеграции с CAD системами.
   - Экспорт модели в указанные форматы.
   - Обеспечить возможность обмена геометрическими данными с внешними CAD-приложениями через стандартные форматы файлов.
3.6.7. Предусмотреть возможность создания параметрических моделей с исследованием влияния входных параметров на результаты (фронт Парето).
3.6.8. Реализовать встроенные библиотеки:
   - Библиотека материалов с возможностью расширения пользователем.
   - Библиотека сечений, включая стандартные сортаменты (сталь, алюминий, дерево) по различным национальным нормам.

3.7. Сквозная функциональность и многозадачность
3.7.1. Обеспечить возможность копирования выбранных элементов (узлов, линий, пластин) из одного открытого окна модели и вставки их в другое окно с другой моделью.
3.7.2. Архитектура программного продукта должна обеспечивать масштабируемость от моделей малого размера (10 точек) до моделей среднего размера (10000 точек и 500 линий) без потери функциональности.
3.7.3. Реализовать систему постоянной проверки данных и валидации модели в процессе ввода с мгновенным уведомлением пользователя об обнаруженных ошибках и несоответствиях.

4. Требования к результатам разработки (проектирование интерфейсов и структур данных)

4.1. В результате выполнения работы должны быть разработаны и представлены интерфейсы и структуры данных на языке Golang, описывающие архитектуру препроцессора. Реализация алгоритмов, внутренней логики и тестирование на данном этапе не требуется.

4.2. Состав разрабатываемых интерфейсов и структур:
4.2.1. Отображение (view):
   - Интерфейсы для рендеринга 3D сцены с использованием OpenGL.
   - Интерфейсы для управления камерой и различными видами.
   - Интерфейсы для отрисовки табличных данных.
   - Интерфейсы для создания и управления панелями инструментов.
   - Интерфейсы для формирования и обработки контекстных меню.
4.2.2. Хранение данных (model):
   - Структуры для представления узлов (node) с координатами X, Y, Z и атрибутами.
   - Структуры для представления линейных элементов (line) со связями с узлами.
   - Структуры для представления треугольных элементов (triangle) со связями с узлами.
   - Структуры для представления четырехузловых пластин (4-node plate) со связями с узлами.
   - Структуры для организации логических групп элементов.
   - Структуры для хранения данных о материалах.
   - Структуры для хранения данных о сечениях.
   - Структуры для представления нагрузок, приложенных к элементам.
   - Структуры для представления комбинаций нагрузок.
   - Структуры для хранения граничных условий.
   - Интерфейсы для доступа и управления всеми типами данных модели.
4.2.3. Печать и экспорт данных:
   - Интерфейсы для сохранения модели во внутренний текстовый формат.
   - Интерфейсы для экспорта модели в форматы DXF и IGES.
   - Интерфейсы для генерации отчетов по модели.
4.2.4. Изменение модели:
   - Интерфейсы для реализации команд с возможностью отмены (Undo/Redo).
   - Интерфейсы для операций создания, редактирования, копирования и вставки элементов.
   - Интерфейсы для выполнения групповых операций над выделенными объектами.
   - Интерфейсы для параметрических изменений модели.
4.2.5. Объединение 3D моделей:
   - Интерфейсы для механизма слияния данных из двух независимых моделей в одну.
   - Интерфейсы для обработки конфликтов при слиянии (совпадающие идентификаторы, дублирующиеся узлы).
4.2.6. Управление проектом:
   - Интерфейсы для работы с несколькими открытыми моделями одновременно.
   - Интерфейсы для управления файлами проектов (открытие, сохранение, закрытие).

4.3. Дополнительные проектные материалы:
4.3.1. Описать предполагаемый пользовательский сценарий для следующих типовых задач:
   - Создание нового узла в графическом режиме.
   - Создание линейного элемента между существующими узлами.
   - Создание поверхностного элемента (пластины) по четырем узлам.
   - Редактирование координат узлов через табличный редактор.
   - Создание нового первичного загружения.
   - Формирование комбинации нагрузок на основе созданных загружений.
   - Объединение двух различных моделей в одну.
4.3.2. Предложить структуру таблиц (перечень колонок) для редактирования узлов и каждого типа элементов с учетом различных атрибутов (координаты, ориентация, местные оси, связи).
4.3.3. Описать интеграцию системы групп с механизмами видимости, выбора и блокировки объектов.
4.3.4. Описать механизм работы с единицами измерения, включая их хранение и конвертацию на лету.
4.3.5. Предложить структуру библиотек материалов и сечений с возможностью расширения пользователем.
4.3.6. Описать структуру дерева нагрузок и комбинаций с указанием необходимых полей и атрибутов для каждого элемента дерева (тип загружения, коэффициенты, статус, видимость).

5. Ограничения и допущения
5.1. При разработке интерфейсов и структур данных не требуется углубляться в вопросы производительности, однако следует предусмотреть архитектурные возможности для будущей оптимизации, включая потенциал использования GPU-вычислений.
5.2. Разработка модульных или интеграционных тестов на данном этапе не требуется.
5.3. Основной фокус разработки направлен на интерфейсы и структуры данных, необходимые для реализации описанной функциональности, без реализации внутренней логики.
