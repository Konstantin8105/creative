ПРАВИЛА РАБОТЫ С ПОЧТОЙ

- Каждое письмо или служебная команда должны быть оформлены как JSON-объект.
- ВАЖНО: Ответ должен быть строго валидным JSON без лишнего текста. Ответ всегда должен быть JSON-массивом.
- Список допустимых получателей для поля "to" передаётся отдельно вне этих правил.
- Для ответа на существующее письмо обязательно указывай его идентификатор в поле "id".
- Для нового письма поле "id" указывать нельзя.
- За один ответ можно отправлять несколько объектов, объединённых в один JSON-массив.
- Поле "to" должно содержать ровно одно имя получателя из списка допустимых ролей.
- Нельзя отправлять письма самому себе. Имя отправителя определяется автоматически системой.
- Если нет необходимости отвечать на входящие письма, можно отправить новое письмо с полезным анализом, уточнением, вопросом или предложением по общей задаче.
- Ограничение: не более 20 объектов за один ответ.
- Предпочтительно сначала ответить на входящие письма, требующие ответа, а затем отправлять новые письма.
- Выбирай получателя по его роли, компетенции и зоне ответственности.
- Не нужно искусственно заполнять лимит. Отправляй только полезные и содержательные письма.
- Перед отправкой ответа проверь, не отвечал ли ты уже на это письмо. Повторный ответ с тем же "id" допустим только если он содержит новую полезную информацию, новую точку зрения или важное уточнение.
- Не отправляй один и тот же вопрос одному и тому же получателю повторно без необходимости. Допускается отправлять схожие вопросы разным получателям, если это оправдано их ролями.
- Если нужно отправить список задач, используй отдельное письмо для каждого пункта.
- В поле "body" не следует ссылаться на номера других писем. Если нужен контекст, кратко перескажи его текст своими словами.
- Важно не только инициировать новые полезные письма, но и своевременно отвечать на входящие письма.
- В поле "to" используй только имена из переданного списка допустимых ролей. Не придумывай новые имена и записывай их точно. Поле "from" заполняется системой автоматически.
- Нельзя выдумывать "id". Используй только реальные идентификаторы, доступные во входных данных. Для новых писем "id" назначается системой автоматически.
- Формулируй письмо с учётом роли получателя так, чтобы оно помогало ему принять решение или продвинуть задачу дальше.
- Все письма должны быть уважительными и деловыми по тону.

Для отправки писем используется следующий формат:

```json
[
  {
    "id": 123,
    "to": "имя_получателя",
    "body": "текст сообщения"
  }
]
```

И отдельно пояснение:
Для нового письма поле id не указывается.
Для ответа поле id обязательно.
Поля to и body обязательны для письма.

Пример нового письма:

```json
[
  {
       "to": "critic",
       "body" : "Предлагаю учесть, как это будет реализовано при программировании."
  }
]
```

Пример ответа на письмо с номером 42:

```json
[
  {
       "id": 42,
       "to": "dreamer",
       "body" : "На мой взгляд, добавь больше идей для разбиения задачи на подзадачи."
  }
]
```

Пример ответа на письмо с номером 65342:

```json
[
  {
       "id": 65342,
       "to": "writer",
       "body" : "По данному thread по вопросу версионности достигнута согласие между всеми и переношу в список решенных задач."
  }
]
```

Пример служебной команды архивирования письма с номером 234. Такую команду может использовать только роль, которой это разрешено. Поле "id" обязательно:

```json
[
  {
    "id": 234,
    "archived": true
  }
]
```

Пример служебной команды перевода письма в состояние solved для письма с номером 345. Такую команду может использовать только роль, которой это разрешено. Поле "id" обязательно:

```json
[
  {
       "id": 345,
       "solved": true
  }
]
```

Перевод письма в состояние solved автоматически переводит письма в архивное письмо.

Пример нескольких отдельных писем по разным темам:

```json
[
  {
    "to": "critic",
    "body": "Броскает риск, что мечтательский дизайн не учтет ограничения ОС Windows. Как убедиться, что UI будет работать на всех версиях?"
  },
  {
    "to": "realist",
    "body": "Нужно уточнить, сколько оперативной памяти понадобится для рендеринга 3D‑модели из 10 000 элементов. Это влияет на тайминги разработки."
  },
  {
    "to": "dreamer",
    "body": "В твоих идеях часто упоминаются «грандиозные» графические эффекты, но они могут потребовать OpenGL 4.5+. Какой уровень совместимости нам нужен?"
  },
  {
    "to": "arxiv",
    "body": "Требуется разделить письмо о планах на две части: часть 1 – цели,часть 2 – задачи. Архивировать оригинал."
  },
  {
    "to": "solved",
    "body": "После обсуждения с критиком и реалістом задача «выбор OpenGL версии» завершена."
  },
  {
    "to": "critic",
    "body": "Как избежать утечек памяти при динамическом добавлении/удалении узлов? Нужно ввести менеджер памяти."
  },
  {
    "to": "arxiv",
    "body": "Разделяем письмо о настройках OpenGL на 2 части: 1) установка драйверов, 2) конфигурация контекста."
  }
]
```

## 1. Правило приоритета входящих писем
`- Если есть письма, требующие срочного ответа, сначала отвечай на них, а уже потом отправляй новые письма.`

Почему полезно:  
помогает не распыляться на новые темы, когда есть незакрытые входящие.

---

## 2. Правило неответа без смысла
`- Не отвечай на письмо, если твой ответ не добавляет новой информации, решения, вопроса, уточнения или полезной точки зрения.`

Почему полезно:  
убирает пустые ответы вроде “согласен”, “ок”, “понял”.

---

## 3. Правило явной цели письма
`- Каждое письмо должно содержать одну понятную цель: вопрос, предложение, уточнение, анализ, решение или запрос действия.`

Почему полезно:  
делает письма структурнее и облегчает дальнейшую обработку.

---

## 4. Правило краткости
`- Письмо должно быть настолько коротким, насколько это возможно без потери смысла.`

Почему полезно:  
снижает шум и риск многотемности.

---

## 5. Правило завершенности мысли
`- Текст письма должен быть самодостаточным и понятным без необходимости искать дополнительные пояснения в других письмах.`

Почему полезно:  
хорошо сочетается с запретом ссылаться на номера других писем.

---

## 6. Правило выбора одного лучшего адресата
`- Если один и тот же вопрос можно отправить нескольким получателям, выбирай того, чья роль наиболее подходит для ответа.`

Почему полезно:  
уменьшает дублирование и лишнюю рассылку.

---

## 7. Правило недопустимости массового дублирования
`- Не отправляй один и тот же текст нескольким получателям без явной необходимости.`

Почему полезно:  
дополняет текущее правило про одинаковые вопросы, но делает его строже.

---

## 8. Правило явного действия в письме
`- Если письмо предполагает реакцию получателя, сформулируй её явно: что именно нужно оценить, решить, уточнить или сделать.`

Почему полезно:  
делает письма практичными, а не абстрактными.

Пример хорошей формулировки:  
“Оцени, пожалуйста, риски по памяти для модели из 10000 элементов.”

---

## 9. Правило проверки на дубли тем
`- Перед отправкой нового письма проверь, не обсуждается ли уже эта тема в другом активном письме.`

Почему полезно:  
помогает избегать параллельных дублей.

---

## 10. Правило аккуратного использования команд
`- Служебную команду "archive" или "solve" используй только тогда, когда для этого есть достаточные основания и это не требует дополнительного обсуждения.`

Почему полезно:  
защищает от преждевременного закрытия писем.

---

## 11. Правило solved только при явной завершенности
`- Команду "solve" используй только если вопрос действительно решён, согласован или больше не требует работы.`

Почему полезно:  
делает статус `solve` более строгим и полезным.

---

## 12. Правило archive только для неактуального
`- Команду "archive" используй только для писем, которые больше не требуют активной обработки, но не обязательно являются решёнными.`

Почему полезно:  
разводит смысл `archive` и `solve`.

---

## 13. Правило корректности JSON перед отправкой
`- Перед формированием ответа проверь, что итоговый JSON валиден, все строки экранированы корректно, а массив и объекты закрыты правильно.`

Почему полезно:  
очень важно для production.

---

## 14. Правило обязательности полезной нагрузки
`- Каждый объект в массиве должен содержать самостоятельную ценность; не добавляй объекты ради количества.`

Почему полезно:  
особенно полезно, если кто-то позже вернёт минимальный порог числа писем.

---

## 15. Правило согласованности стиля
`- Используй нейтральный, уважительный и профессиональный стиль без эмоциональных оценок, если они не нужны для смысла.`

Почему полезно:  
делает коммуникацию стабильной.

---

## 16. Правило отказа от предположений
`- Не утверждай факты, которых нет во входных данных. Если информации недостаточно, задай уточняющий вопрос.`

Почему полезно:  
сильно уменьшает галлюцинации в переписке.

---

## 17. Правило явного разделения факта и мнения
`- Если пишешь предположение, анализ или мнение, формулируй это как предположение, анализ или мнение, а не как установленный факт.`

Почему полезно:  
повышает качество коллективной работы агентов.

---

## 18. Правило проверки адресата
`- Перед отправкой письма проверь, что выбранный получатель действительно подходит по роли для темы письма.`

Почему полезно:  
защищает от ошибок маршрутизации.

---

## 19. Правило неотправки конфликтующих команд
`- Не отправляй для одного и того же "id" в одном ответе конфликтующие команды или несовместимые действия.`

Почему полезно:  
например, чтобы не было одновременно `archive` и `solve` без нужды.

---

## 20. Правило уникальности объектов в одном массиве
`- В одном ответе не должно быть полностью одинаковых объектов.`

Почему полезно:  
простое и полезное ограничение против случайного дубля.

---
